# 上下文翻譯 設計文件

日期：2026-07-03
狀態：已與使用者確認方向
關聯：與《模組名譯名與自訂用語庫》（`2026-07-03-modname-glossary-design.md`）同日設計，
共用其第 7 節的 prompt 組裝約定；兩者可獨立實作，建議用語庫先行。

## 問題與目標

1. **鄰近文本語境**：任務標題與描述被分批邏輯盲切到不同請求，模型看不到完整任務，
   代名詞指涉與敘事連貫性差。
2. **同輪用語一致**：同一個詞（尤其專有名詞式短語）在包內前後被翻成不同譯法。
3. **每包語境**：不同模組包的題材/語氣不同，使用者希望能描述包的世界觀讓模型參考，
   且記憶（譯法、描述）要按包分開、跨執行留存。

成本約束：使用者用付費遠端 API，鄰近語境採**近零成本策略**（重排本來就要送的內容，
不額外附帶參考文本）。

## 1. 每包記憶：`<模組包>/.modpack_translator/context.json`

```json
{
  "extra_prompt": "這是魔法主題包，語氣輕鬆口語…",
  "learned_terms": { "Starlight Sanctum": "星輝聖所" }
}
```

- 存放在模組包資料夾內。理由：工具本來就寫入包資料夾（`mods_bak/`、`quests_bak/`），
  天然一包一份，不需發明包的身分識別；包搬家/換電腦/多份安裝，記憶跟著包走。
- 執行開始時載入，翻譯結束（含取消）時寫回。缺檔、壞檔視為空記憶，不報錯。
- GUI 選到包路徑時自動載入 `extra_prompt` 帶回介面。

## 2. 鄰近語境 = 智慧分批（近零成本）

現況：批次預翻譯照收集順序固定每 12 條切一批，任務的 title/desc 可能被切開。

改為批次邊界對齊語義邊界：

- 同一任務的字串（`quest.HEX.*` 的 title/subtitle/quest_desc 等，共同前綴 `quest.HEX`）
  視為一組，保證同批不被拆開；模型一次看到完整任務，敘事與代名詞自然連貫。
- 一批可包含多個組（含不同檔案的組）——維持填充率，請求數與成本不增加。
  若強制檔案邊界封批，大量只有幾條字串的模組 lang 檔會產生零碎批次，反而增加請求數。
- 批次大小由固定值改為彈性：目標大小沿用設定值（預設 12），為容納整組允許
  溢出至 4/3 倍（預設 16 為硬上限）；單組超過硬上限或字元預算的罕見情況拆組。
- 額外 token 成本為零——只是重排本來就要送的字串。
- 僅影響遠端批次預翻譯路徑；本機逐條路徑無分批概念，不變。

## 3. 同輪用語一致 = 動態用語庫（第四層，權限受限）

- **記錄條件**：整串翻譯成功時，原文為專有名詞式短語（沿用
  `_looks_like_proper_noun_phrase` 判定）且譯文與原文不同、含 CJK → 記入
  `learned_terms`。只記「整串對整串」配對，不做句中對齊（會猜錯）。
  自訂/模組名/官方用語庫已涵蓋的詞不重複記錄。
- **使用**：只參與 prompt 注入（`[Glossary]` 區塊），優先序最低
  （自訂 > 模組名 > 官方 > 動態）。
- **不給的權限**：不參與 `enforce` 事後替換、不參與三入口守門——機器自學的譯法
  若一開始就錯，給強制力會把錯誤確定性地擴散到整包。修正途徑：自訂用語庫優先序
  更高可蓋掉，或直接編輯包內 JSON。
- **實作**：獨立的第二個 Glossary 實例（injection-only），不混入三層合併的主用語庫，
  權限差異在物件層面分開。`augment_prompt` 合併兩個實例的命中結果時，
  詞數上限內由主用語庫優先佔用。
- **同輪即時性**：執行中共享 dict 隨批次/單條完成即時更新（加鎖），
  後續請求立刻受益；已在飛行中的併發批次拿不到，屬盡力而為。

## 4. 額外提示詞（自由文字框）

- 組裝順序照用語庫 spec 第 7 節約定：
  `system prompt（靜態）→ 額外提示詞（整輪不變，靜態段）→ [Glossary]（動態，最後）`。
  實作：建立 translator 前把 `extra_prompt` 併入 system_prompt 字串，
  用語庫機制對此無感知，prompt cache 前綴穩定。
- 每包記憶：存 `context.json`，換包自動切換。
- 分工界線（GUI 文案要講清楚）：詞彙對應（X 譯為 Y）請進自訂用語庫
  （才有短路省費、強制一致）；這裡放題材/語氣/受眾描述。

## 5. GUI

主視窗淨增量：兩個按鈕＋一個 checkbox（含用語庫 spec 的部分），全部收進
選項群組收斂後的兩列，淨高度仍比現況少一列（垂直空間約束見用語庫 spec 第 4 節）。

- 「翻譯語境…」按鈕開 QDialog：多行文字框（`extra_prompt`）＋分工說明文案。
  按鈕文字反映狀態：「翻譯語境…（已設定）」。
- 儲存寫回目前所選包的 `context.json`；未選包時按鈕停用。
- `learned_terms` 不做編輯介面（修正途徑走自訂用語庫），保持 GUI 簡單。

## 6. 測試

- 智慧分批：同任務不跨批、同檔不跨批、批大小上下限、超大組拆分、
  空輸入/單條輸入邊界。
- learned_terms：記錄條件（專有名詞式才記、含 CJK 才記、已在主用語庫的不記）、
  併發寫入安全、寫回與載入 roundtrip、壞檔容錯。
- 權限隔離：動態層條目不觸發 enforce、不觸發三入口守門（回歸測試各一）。
- 額外提示詞：插入位置在 `[Glossary]` 之前、空字串時完全不影響 prompt、
  換包後載入正確的 extra_prompt。

## 風險與取捨

- 動態用語庫學到不理想的譯法會在同包後續請求中被建議（僅建議，非強制）；
  接受此風險以換取一致性，且自訂用語庫可蓋掉。
- 彈性批次大小改變既有分批節奏，吞吐略受影響（批平均變小則請求數略增）；
  以 8–16 區間控制，預期影響很小。
- `.modpack_translator/` 寫入包資料夾，發佈整合包時使用者可能連同記憶檔一起打包；
  屬無害的純文字檔，文件註明即可。
